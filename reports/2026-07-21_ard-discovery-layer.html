ARD — AI 에이전트 발견계층의 길목

구글이 공개한 에이전트 기능 발견·검증 개방 표준. "찾고(검색) + 진짜인지 확인(신원)"의 공통 규칙과, 그 안에 숨은 길목 권력.

관찰 시점 2026-06-17 정리 2026-07-21 개방 표준 · Apache 2.0
  • 한 줄ARD(Agentic Resource Discovery) = AI 에이전트들이 서로의 도구·기능·다른 에이전트를 인터넷에서 찾고(검색) + 진짜인지 확인(검증)하는 공통 규칙. 조직 경계를 넘어 capability(기능)를 발견·검증·연결하는 개방형 명세다.
  • 발표구글 개발자 블로그, 2026-06-17. 라이선스는 Apache 2.0, 기반 데이터 모델은 Linux Foundation의 "AI Catalog".
  • 2 부품카탈로그(회사가 자기 도메인에 올리는 메뉴판) + 레지스트리(그 메뉴판을 크롤링·색인하는 "에이전트 세상의 검색엔진").
  • 위치MCP·A2A가 "연결 후 대화하는 법"이라면, ARD는 그 이전 단계 — "상대를 찾고 진짜인지 확인하는 법". 경쟁이 아니라 다른 층이다.
  • 🔴 핵심발견 계층의 진짜 권력은 표준 문서가 아니라 "레지스트리를 누가 쥐느냐". 규격은 열되 검색 길목은 선점하려는 구도로 읽힌다.

1. ARD가 풀려는 문제 — 에이전트의 세 가지 질문

한 AI 에이전트가 자기 조직 밖의 도구·기능·다른 에이전트를 쓰려면 세 가지를 먼저 알아야 한다.

질문지금까지ARD가 주는 것
① 그게 어디에 있나회사마다 자기 registry를 따로 운영 → 서로 못 넘는 칸막이(silo)조직을 넘는 공통 발견 규칙
② 어떤 걸 써야 하나평범한 말로 검색해 매칭되는 capability를 받음
③ 연결해도 안전한가연결 전에 "진짜 그 회사 맞나"를 암호학적으로 검증

즉 ARD는 흩어진 registry의 칸막이를 걷어내고, 조직을 넘는 발견·신뢰 계층을 하나 얹는 시도다.

2. 핵심 부품 2개 — 카탈로그와 레지스트리

카탈로그 (Catalog)

회사가 자기 도메인의 정해진(well-known) 경로에 ai-catalog.json을 게시하는 방식이다. 그 안에 MCP 서버·A2A 에이전트·OpenAPI 도구·중첩 카탈로그의 목록이 담긴다. 핵심은 도메인 소유권 자체가 암호학적 신원·신뢰의 뿌리라는 점 — 그 도메인의 주인만 게시할 수 있으므로, "누가 발행했는가"가 DNS처럼 자연히 보증된다.

레지스트리 (Registry)

카탈로그들을 크롤링·색인하는 "에이전트 세상의 검색엔진"이다. 에이전트가 평범한 말(plain-language intent)로 검색하면, 매칭되는 capability와 함께 발행자 검증 메타데이터를 돌려준다. 발견의 편의는 여기서 나오고, 뒤에서 다룰 권력 문제도 바로 이 지점에 걸린다.

3. 작동 4단계 — 검색은 경유, 연결은 직접

주목할 설계는 검색과 연결의 분리다. 찾는 과정(②)은 레지스트리를 거치지만, 신원 검증(③)이 끝나 실제로 통신을 시작하면(④) ARD는 빠지고 도구 고유의 native 규격으로 제공자와 직접 연결한다. 레지스트리는 "찾아주는" 역할까지만 하고, 오가는 실제 데이터의 길목에는 앉지 않는다 — 이 점이 뒤의 권력 논의를 미묘하게 만든다.

4. MCP·A2A와의 관계 — 경쟁 아닌 다른 층

흔한 혼동은 ARD를 MCP나 A2A의 경쟁 규격으로 보는 것이다. 실제로는 층이 다르다.

ARDMCP · A2A
역할상대를 찾고 진짜인지 확인하는 법연결한 뒤 대화하는 법
비유전화번호부 + 발신자 진위 확인실제 통화
시점연결 이전연결 이후

검증이 끝나면 ARD는 물러나고, 그다음부터는 MCP·A2A·OpenAPI로 직접 대화한다. 발견 계층(ARD)과 대화 계층(MCP·A2A)은 서로를 대체하는 것이 아니라 위아래로 쌓인다.

5. 구글 제품과의 연결 — Agent Registry

표준의 기업용 구현체로는 Gemini Enterprise Agent Platform의 "Agent Registry"가 제시됐다. 고유 URN 부여, egress(외부 전송) 정책, 도구 pinning(고정), Agent Identity 기반 trust manifest 검증, HIPAA 준수 같은 거버넌스 장치를 갖췄고, ARD 네이티브 지원은 "앞으로 몇 달 내"로 예고됐다.

여기서 구도가 드러난다. 개방 규격(ARD)은 누구나 쓰도록 열어두면서, 그 규격을 실제로 굴리는 검색엔진(Agent Registry)은 자사 플랫폼으로 밀고 있다. 규격과 길목이 분리돼 움직인다.

6. 구조적 함의 — 진짜 권력은 "누가 레지스트리를 쥐느냐"

ARD는 이른바 "agentic web(에이전트들이 서로를 발견·연결하는 웹)"의 발견·신뢰 계층을 표준화하려는 시도로 볼 수 있다. 다만 표준을 발표한 것과 표준이 정착하는 것은 다른 문제다.

과거 웹에서 HTML 문법 자체는 공짜였지만, "무엇이 어디 있는지 찾아주는" 검색 길목의 권력은 소수에게 집중됐다. ARD도 같은 구조를 품고 있다 — 규격 문서는 열려 있어도, 발견 계층의 실질 권력은 카탈로그를 크롤링·색인하는 레지스트리를 누가 쥐느냐에서 갈린다. 구글이 규격을 개방형으로 내면서 동시에 Agent Registry를 미는 것은 "규격은 열되 길목은 선점"하는 포석으로 읽을 수 있다.

결론적으로, 곡괭이·병목 렌즈로 보면 레지스트리 = 발견 계층의 길목(통행료) 후보다. 그러나 앞서 본 설계상 실제 연결은 레지스트리를 우회하므로, 길목의 두께가 검색 유통의 통제만큼 깊을지는 아직 열린 문제다. 채택이 여러 레지스트리로 분산될지, 소수에 집중될지가 이 표준의 실제 무게를 결정한다. 정리하자면 ARD는 판을 비추는 거울이지, 특정 종목의 직접 노출로 곧장 연결되는 신호는 아니다 — 추적할 질문은 하나로 압축된다: "레지스트리는 누가 쥐는가."
